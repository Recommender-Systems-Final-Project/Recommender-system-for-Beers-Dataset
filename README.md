## Overview

Multi-member team project building a beer-recommendation system from **~3.3 million reviews** scraped from BeerAdvocate and RateBeer. The codebase is organised into two phases:

| Phase | Goal | Notebooks |
|---|---|---|
| **Phase 1** | Shared baselines every variant compares against | Item-based CF, Matrix Factorisation |
| **Phase 2** | Individual *advanced variants* — each owner extends Phase 1 with a different method | Content-Boosted Item-CF, NeuMF, Hybrid NeuMF + MetadataSASRec, LightGCN graph model |

Everything is evaluated under a **single shared protocol** (`K = 10`, leave-last-out, K-core filtering, `seed = 42`) so the variants are directly comparable.

## Dataset

| Source | Records | Format | Size (gz) |
|---|---|---|---|
| BeerAdvocate | ~1.6 M reviews | one Python-dict literal per line | ~448 MB |
| RateBeer | ~1.7 M reviews | one Python-dict literal per line | ~399 MB |

> Both corpora come from the [Stanford SNAP](https://snap.stanford.edu/data/) Beer Reviews datasets.

**Rating normalisation.** BeerAdvocate ratings are 0–5 floats, divided by 5 to land in `[0, 1]`. RateBeer ratings come as fractions like `"13/20"` — parsed and converted to `[0, 1]`.

The two are tagged with a `source` column and concatenated into `df_combined.parquet`, which every downstream notebook consumes.

## Methods

### Phase 1 — shared baselines

**Item-based Collaborative Filtering** *(`common/Beer_Recommender_Phase1_ItemCF.ipynb`)*
Builds a sparse user × item rating matrix, computes cosine item-item similarity with shrinkage, and predicts a user's rating on a candidate item as the similarity-weighted average over items the user has already rated. Top-K aggregation; HR@K / NDCG@K under leave-last-out.

**Matrix Factorisation** *(`common/Beer_Recommender_Phase1_MF_B1.ipynb`)*
Latent-factor model in PyTorch:

```
ŷ(u, i) = μ + b_u + b_i + p_uᵀ q_i
```

Trained with MSE on observed ratings; reports RMSE plus the same ranking metrics so it can be compared against ItemCF and Phase 2.

### Phase 2 — advanced variants

#### A. Content-Boosted Item-CF
`variants/cb_itemcf/Beer_Recommender_Phase2_ItemCF.ipynb`

Keeps the Phase 1 ItemCF *scoring rule* but **fuses three item-item similarities**:

```
S(i, j) = α · S_int(i, j)  +  β · S_meta(i, j)  +  γ · S_sem(i, j)
                                  α + β + γ = 1
```

| Channel | Source | Captures |
|---|---|---|
| `S_int` | interactions (cosine on user × item matrix) | "users who liked X also liked Y" |
| `S_meta` | structured metadata (style, brewery, ABV, sensory aspects) | shared category & taste profile |
| `S_sem` | SBERT embeddings (`all-MiniLM-L6-v2`) over text | semantic / descriptive similarity |

α / β / γ are grid-searched on a held-out split. Includes a **cold-item slice** so cold performance is reported separately.

#### B. Neural Collaborative Filtering (NeuMF)
`variants/neumf/Phase2_NeuMF.ipynb`

End-to-end pipeline: data load → preprocessing → MF baseline → **GMF + MLP fusion** (He et al., *WWW 2017*) → ranking eval. The MLP tower learns non-linear user-item interactions while GMF preserves the bilinear inner-product term.

#### C. HybridNeuMF + MetadataSASRec
`variants/hybrid_sasrec/Beer_Recommender_Hybrid_SAS.ipynb`, `variants/hybrid_sasrec/NeuMF_SASRec.ipynb`

Two architectures in one pipeline (the folder holds two iterations of the notebook — `NeuMF_SASRec.ipynb` is the latest):

- **HybridNeuMF** — NeuMF augmented with beer side-information (style, brewery, ABV, sensory aspects).
- **MetadataSASRec** — self-attentive sequential recommender (Kang & McAuley, *ICDM 2018*) extended with the same metadata block.

On top of training, the inference layer adds:

- **Bayesian cold-start fallback** — for users with sparse history, posterior interpolation between content affinity and global popularity.
- **MMR re-ranking** — Maximal Marginal Relevance to enforce style diversity in the Top-N.

#### D. LightGCN Graph Recommender
`variants/lightgcn/graph_recommender.py`, `variants/lightgcn/evaluate_graph.py`

Pure-Python (script-form) implementation of **LightGCN** (He et al., *SIGIR 2020*) on the user-item bipartite graph:

```
E^(k) = D^(-1/2) · A · D^(-1/2) · E^(k-1)
E_final = mean(E^(0), E^(1), …, E^(K))
```

Trained with **BPR loss** (Bayesian Personalized Ranking — pairwise positive/negative sampling). A separate **ContentEncoder** projects beer style + brewer + ABV into the embedding space so cold items still get a meaningful score. `evaluate_graph.py` produces:

- Full test-set NDCG / Recall / Precision / MRR at multiple K
- Learning curves (train loss + validation NDCG)
- Cold-user vs warm-user metric breakdown
- Coverage & diversity
- Per-style recommendation analysis

## Evaluation protocol

Fixed across **every** notebook — this is what makes the variants comparable.

| Constant | Value | Meaning |
|---|---|---|
| `K` | `10` | top-K for headline metrics (also reported at 5, 20) |
| `SEED` | `42` | reproducibility |
| `RELEVANCE_THRESHOLD` | `0.7` | normalised rating cutoff for "positive" |
| `MIN_USER_INTERACTIONS` | `5` | drop users with < 5 ratings |
| `MIN_ITEM_RATES` | `5` | drop items with < 5 ratings |
| `KCORE_MAX_PASSES` | `5` | iterative K-core pruning ceiling |
| `EVAL_USER_SAMPLE` | `1000` | sampled users at full eval |
| `GRID_EVAL_SAMPLE` | `500` | sampled users during hyper-parameter grid search |
| `TOP_NEIGHBORS` | `200` | neighbour cap for ItemCF / CB-ItemCF |
| `SBERT_MODEL` | `sentence-transformers/all-MiniLM-L6-v2` | SBERT encoder for `S_sem` |

Splits are **leave-last-out** (chronological where applicable), with HR@K and NDCG@K as the primary ranking metrics; RMSE additionally for rating-prediction models.

## How to run

### 1. Environment

```bash
python -m venv venv
source venv/bin/activate

pip install \
    numpy pandas scipy scikit-learn matplotlib seaborn \
    torch \
    sentence-transformers==3.0.1 \
    pyarrow nbformat
```

> **Colab.** The notebooks auto-detect Colab via `'google.colab' in sys.modules`. Mount Drive and point `BASE_PATH` at your data folder; nothing else changes.

### 2. Data

Drop `beeradvocate.json.gz` and `ratebeer.json.gz` inside `datasets/`, then run:

```
common/EDA.ipynb        →  produces datasets/df_combined.parquet
```

### 3. Pick your path

| You want to … | Run this |
|---|---|
| Reproduce **Phase 1** results | `common/Beer_Recommender_Phase1_ItemCF.ipynb`, `common/Beer_Recommender_Phase1_MF_B1.ipynb` |
| Reproduce **CB-ItemCF** | `variants/cb_itemcf/Beer_Recommender_Phase2_ItemCF.ipynb` |
| Reproduce **NeuMF** | `variants/neumf/Phase2_NeuMF.ipynb` |
| Reproduce **HybridNeuMF + SASRec** | `variants/hybrid_sasrec/NeuMF_SASRec.ipynb` (latest) or `Beer_Recommender_Hybrid_SAS.ipynb` |
| Reproduce **LightGCN** | `python variants/lightgcn/graph_recommender.py` then `python variants/lightgcn/evaluate_graph.py` |
| **Skip training** & only see metrics | `common/Beer_Recommender_LoadAndEval.ipynb` (loads `common/*.pth` + `common/baselines.pkl`) |

> **Path constants — heads up.** Notebook path-config cells were written when files lived at the repo root. Open the first config cell of any notebook you run and update its `PROJECT_DIR` / `BASE_PATH` to point at the correct subfolder (e.g. raw data is now under `datasets/`, weights under `common/`).

### 4. LightGCN — script invocation

```bash
# train
DATA_DIR=./datasets python variants/lightgcn/graph_recommender.py

# evaluate (after training saved lightgcn_model.pt)
DATA_DIR=./datasets python variants/lightgcn/evaluate_graph.py
```

## Headline results

| Variant | Strength | Notes |
|---|---|---|
| Phase 1 ItemCF | strong simple baseline | reference for every Phase 2 lift |
| Phase 1 MF | learned embeddings | RMSE + ranking dual report |
| **Phase 2 CB-ItemCF** | beats ItemCF on HR@10 & NDCG@10 | (α, β, γ) grid-searched; cold-item slice reported separately |
| **Phase 2 NeuMF** | non-linear user-item interaction | GMF + MLP towers fused |
| **Phase 2 HybridNeuMF** | ≈ 9.8 % RMSE improvement on cold items vs. global mean | side-info embedding |
| **Phase 2 SASRec + MMR** | ≈ 90 % style diversity in Top-10 | Bayesian fallback narrowed cold-user gap from −44.4 % to −4.8 % vs popularity |
| **Phase 2 LightGCN** | high-order neighbourhood signal | BPR loss; cold items via content encoder |

> Exact numbers per notebook live in each file's final evaluation cell. `common/Beer_Recommender_LoadAndEval.ipynb` aggregates them in one place.
